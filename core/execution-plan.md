# Execution Plan Format

The Execution Plan is the master blueprint for any complex workflow within the PEN.GUIN ecosystem. It is a strictly formatted, machine-readable document generated by the `Planner Agent` and consumed by the `AI Kernel` to orchestrate multi-agent collaboration.

## Plan Generation and Consumption

1. **Generation:** When a user submits an objective, the `Planner Agent` analyzes the request, maps the required changes against the current workspace, and decomposes the work into a structured Execution Plan (JSON or YAML format). 
2. **Consumption:** The `Planner Agent` submits this plan to the Kernel via the `submit_plan` protocol. 
3. **Execution:** The `Workflow Engine` parses the plan, calculates the dependency graph (DAG), and translates each `subtask` into individual, trackable files in the `/workspace/tasks/` queue. The `Agent Router` then uses the `assigned_agents` and `required_skills` metadata to provision and distribute the work.

## Structured Plan Format (Schema)

To ensure predictable orchestration, all Execution Plans must adhere to the following structural schema.

```json
{
  "plan_id": "plan-uuid-string",
  "objective": "High-level description of the user's original request.",
  "status": "pending | active | completed | failed",
  
  "subtasks": [
    {
      "task_id": "subtask-1",
      "description": "Clear, actionable directive for a single unit of work.",
      "assigned_agent_type": "frontend-agent | backend-agent | review-agent | etc",
      "competency_tags": ["UI", "React", "State Management"],
      
      "required_skills": [
        "component-generator",
        "css-utility-mapper"
      ],
      
      "dependencies": [
        // List of task_ids that must be completed before this task can begin.
        // e.g., "subtask-0"
      ],
      
      "context_requirements": [
        // Specific files or previous artifacts this agent needs to read.
        "/workspace/docs/api-contracts/user-auth.json"
      ],
      
      "expected_artifacts": [
        {
          "type": "code",
          "path": "/workspace/src/components/LoginForm.jsx",
          "validation_criteria": "Must pass ESLint and a11y-validator."
        }
      ]
    },
    {
      "task_id": "subtask-2",
      "description": "Example dependent task.",
      "assigned_agent_type": "review-agent",
      "competency_tags": ["Security", "SAST"],
      "required_skills": ["sast-analyzer"],
      "dependencies": ["subtask-1"], // Cannot start until subtask-1 is complete
      "context_requirements": ["/workspace/src/components/LoginForm.jsx"],
      "expected_artifacts": [
        {
          "type": "report",
          "path": "/workspace/artifacts/reports/security-scan-subtask-1.md"
        }
      ]
    }
  ]
}
```

### Schema Breakdown

*   **`plan_id` & `objective`:** High-level metadata linking the plan back to the original prompt.
*   **`subtasks`:** The core array of work units.
    *   **`task_id`:** A unique identifier used for mapping dependencies and tracking in the `Task Queue`.
    *   **`description`:** The exact instruction passed to the assigned agent.
    *   **`assigned_agent_type` & `competency_tags`:** Instructions for the `Agent Router` to match the task to the correct specialist.
    *   **`required_skills`:** A pre-computed list of tools the `Skill Router` must provision from the `Skill Registry` before the agent begins.
    *   **`dependencies`:** Crucial for the `Workflow Engine`'s topological sorting. Tasks with dependencies remain "Blocked" until the referenced `task_id` is marked "Completed".
    *   **`context_requirements`:** Instructs the `Context Engine` on exactly which files to inject into the agent's prompt, preventing token bloat.
    *   **`expected_artifacts`:** Defines the strict success criteria. The `Workflow Engine` uses this to verify the agent actually produced the required output before marking the task complete.

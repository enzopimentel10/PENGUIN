# Task Node Specification

A "Task Node" is the fundamental atomic unit of work within the PEN.GUIN ecosystem's `Task Graph`. While an `Execution Plan` defines the overall strategy, a Task Node defines the exact, executable parameters of a single step.

The `Workflow Engine` instantiates these nodes in memory and serializes them as JSON files in the `/workspace/tasks/` directory, where they are consumed by agents.

## Task Node Schema

Every Task Node must conform to the following comprehensive schema to ensure proper routing, execution, and validation.

```json
{
  "task_id": "string (UUID or semantic identifier)",
  "objective": "string",
  "execution_status": "blocked | pending | assigned | active | completed | failed",
  
  "routing_metadata": {
    "assigned_agent": "string (null until assigned by Router)",
    "competency_tags": ["string"],
    "required_skills": ["string"]
  },

  "graph_metadata": {
    "dependencies": ["task_id_string"],
    "dependents": ["task_id_string"]
  },

  "payload": {
    "instruction": "string (The actual prompt given to the agent)",
    "input_artifacts": [
      {
        "type": "string (code | report | context)",
        "path": "string (path to the artifact generated by a dependency)",
        "description": "string (why the agent needs to read this)"
      }
    ],
    "output_artifacts": [
      {
        "type": "string (code | schema | log)",
        "expected_path": "string (where the agent MUST save the result)",
        "validation_criteria": "string (instructions for the Review Agent)"
      }
    ]
  },
  
  "execution_log": {
    "session_id": "string (UUID of the active execution session)",
    "started_at": "timestamp",
    "completed_at": "timestamp",
    "error_trace": "string (null unless execution_status is 'failed')"
  }
}
```

## Field Definitions

### `task_id`
A unique identifier for the node. It is used heavily by the `Task Graph` to map dependencies (edges) and by the `Artifact System` to maintain relational traceability.

### `execution_status`
The state machine of the node:
- `blocked`: Cannot start. Waiting on nodes listed in `dependencies`.
- `pending`: Dependencies resolved. Waiting in the queue for an agent.
- `assigned`: Claimed by the `Agent Router`.
- `active`: Agent is actively running its cognitive loop on this task.
- `completed`: Agent finished, and output artifacts passed validation.
- `failed`: Agent encountered an unrecoverable error or exceeded retry limits.

### Routing Metadata
- `assigned_agent`: Populated dynamically by the `Agent Router` once the node becomes `pending`.
- `required_skills`: A list of skills (e.g., `component-generator`) that the `Skill Router` must mount and provision to the agent's prompt before execution begins.

### Graph Metadata
- `dependencies`: Task IDs that must be `completed` before this node can transition from `blocked` to `pending`.
- `dependents`: Task IDs that are waiting on *this* node. When this node completes, the `Workflow Engine` notifies these dependents.

### Input and Output Artifacts
This is the core of the data-passing mechanism between chained skills and agents.
- `input_artifacts`: Tells the `Context Engine` exactly which files to inject into the agent's prompt. Often, these are the `output_artifacts` of the nodes listed in `dependencies`.
- `output_artifacts`: Defines the strict contract for success. The task is not marked `completed` until files matching these descriptions are verified to exist at the `expected_path` by the `Workflow Engine`.

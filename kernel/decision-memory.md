# Decision Memory System

The Decision Memory System is the historical database of the PEN.GUIN ecosystem. It meticulously records the context, choices, and outcomes of every action taken by the AI Kernel, forming the foundation for the system's continuous learning and predictive routing capabilities.

## How Historical Decisions Are Stored

Instead of unstructured logs, the Decision Memory utilizes a multi-modal storage approach:
- **Vector Embeddings:** Natural language descriptions of tasks and code contexts are converted into vector embeddings. This allows the system to perform semantic similarity searches (e.g., finding past tasks that "sound like" the current one).
- **Relational Graph:** Decisions (Agent X chose Skill Y for Task Z) are stored as a directed graph, linking tasks, agents, skills, and outcomes together.
- **Time-Series Metrics:** Performance scores and execution times are stored in a time-series format to track agent/skill degradation or improvement over time.

## Stored Data Schema

The Decision Memory captures a comprehensive snapshot of every execution cycle, structured into the following categories:

### 1. Task History
- **Initial Prompt & Context:** The raw objective provided to the system.
- **Classification Tags:** The domain and intent tags generated by the `Task Classifier` (e.g., `Frontend`, `Auth`, `Refactor`).
- **Semantic Embeddings:** A vector representation of the task description for fast retrieval.

### 2. Agent Decisions
- **Routing Choices:** A record of which agent the `Agent Router` selected from the available pool and *why* (e.g., based on competency weighting or availability).
- **Cognitive Steps:** Key decision points extracted from the agent's internal `Observe -> Orient -> Decide` loop.

### 3. Skill Selections
- **Provisioned Capabilities:** Which skills the `Skill Router` made available to the agent for the task.
- **Skill Invocations:** The specific skills the agent actually chose to invoke, including the parameters passed and the sequence of chaining.
- **Alternative Paths:** Any skills that were considered by the router but ultimately rejected due to poor historical performance.

### 4. Execution Outcomes
- **Artifact Hashes:** Pointers to the final generated code or reports in `/workspace/artifacts/`.
- **Validation Results:** The boolean pass/fail status from automated test suites or linters.
- **Review Agent Feedback:** Specific critiques, bug reports, or security vetoes generated during the validation phase.

### 5. Performance Metrics
- **Efficiency Data:** Wall-clock execution time, cognitive loop iterations, and API token consumption.
- **Reliability Data:** Error rates, crash reports, and the number of recovery attempts required.
- **Calculated Trust Scores:** The final delta applied to the involved agent's and skill's overall Trust Score.

## Improving Future Routing and Execution

By querying this structured memory, the PEN.GUIN ecosystem transforms past experiences into future optimizations:

- **Predictive Agent Routing:** When a new task arrives, the `Agent Router` queries the vector database for semantically similar past tasks. If a previous task involving "Stripe API Integration" was completed exceptionally well by a specific `Backend Agent`, the router biases its current decision toward that same agent, bypassing standard tie-breaking rules.
- **Proactive Skill Provisioning:** If the memory shows that agents always struggle with a specific framework unless they use the `component-generator` skill, the `Skill Router` will proactively provision that skill the moment the framework is detected in a new task, even if the agent hasn't explicitly asked for it.
- **Automated Guardrails:** If the memory graph indicates a high correlation between the `Refactor Agent` using a specific `ast-manipulator` skill and subsequent test failures, the Kernel will dynamically sever that connection, blocking that specific agent-skill pairing in future executions.
- **Contextual Priming:** The `Context Engine` can retrieve the "Cognitive Steps" of successful past executions and inject them into a new agent's prompt as an example ("Here is how a similar problem was solved previously"), dramatically reducing the number of iterations needed to find a solution.

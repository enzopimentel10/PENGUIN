# Agent Router

The Agent Router is responsible for matching the classified sub-tasks (generated by the Task Classifier) to the most appropriate AI agent within the PEN.GUIN ecosystem.

## How the System Decides Which Agent Should Handle a Task

The routing decision logic relies on the `agent-registry.md` and the tags applied to each sub-task:

1. **Competency Matching:** The router compares the domain tags of the sub-task against the registered competencies of available agents. 
   - A task tagged `Client-Side` and `UI` is routed to the **Frontend Agent**.
   - A task tagged `Server-Side` and `Database` is routed to the **Backend Agent**.
2. **Phase Alignment:** The router considers the current phase of the development lifecycle.
   - Initial planning phases route directly to the **Architecture Agent**.
   - Post-implementation tasks automatically route to the **Test Agent**, **Review Agent**, and **Security Agent**.
3. **Authorization Checks:** The router verifies if the selected agent has the necessary permissions to access the required files or context for the task.
4. **Handoff Coordination:** If a task requires multiple agents in sequence, the Agent Router orchestrates the handoff. It signals the Context Engine to synthesize the output of the departing agent (e.g., an API contract from the Backend Agent) into the starting context for the next agent (e.g., the Frontend Agent).

## Decision Logic Prioritization
If multiple agents could theoretically handle a task, the Agent Router prioritizes the most specialized agent to minimize context overlap and maintain strict architectural boundaries.

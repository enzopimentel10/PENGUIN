# Agent Types and Architecture

The PEN.GUIN ecosystem employs a multi-agent architecture. Instead of a single, monolithic AI trying to do everything, the system divides labor among highly specialized agents, each operating within strict domain boundaries.

## Defined Agent Types

### 1. Architect Agent
- **Role:** The high-level system designer.
- **Responsibilities:** Translates vague user requirements into concrete technical roadmaps. It determines the folder structure, selects the tech stack, defines database schemas, and writes the initial API contracts. It ensures all plans adhere to the `ARCHITECTURE_GUARDRAILS.md`.
- **Interaction:** It does not write implementation code. It produces design artifacts that the Kernel uses to generate sub-tasks for the implementation agents.

### 2. Backend Agent
- **Role:** The server-side implementer.
- **Responsibilities:** Builds the core logic of the application. It implements the API endpoints, writes database queries and migrations, handles authentication/authorization logic, and integrates with third-party services.
- **Interaction:** Relies heavily on the API contracts generated by the Architect Agent. It hands off its completed routes to the Review Agent and the Test Agent.

### 3. Frontend Agent
- **Role:** The client-side implementer.
- **Responsibilities:** Focuses entirely on the user interface and client-side state. It scaffolds components, applies styling (CSS/Tailwind), connects the UI to the backend APIs, and ensures accessibility standards are met.
- **Interaction:** It operates under strict boundaries and is explicitly forbidden from modifying backend logic. It frequently utilizes skills from the `component-generator` and `css-utility-mapper` categories.

### 4. Review Agent
- **Role:** The automated peer reviewer and quality gate.
- **Responsibilities:** Analyzes code generated by other agents. It checks for stylistic consistency (linting), cyclomatic complexity, and adherence to `AI_ENGINEERING_RULES.md`. It also performs static security analysis (SAST) to look for hardcoded secrets or common vulnerabilities.
- **Interaction:** Does not generate new features. If it finds flaws, it rejects the task, providing the original authoring agent with a detailed list of required corrections.

### 5. Documentation Agent
- **Role:** The knowledge maintainer.
- **Responsibilities:** Ensures the project remains comprehensible. It generates and updates `README.md` files, extracts inline code comments into readable API documentation, and maintains the changelog.
- **Interaction:** Typically runs at the end of a development cycle. It relies on the Context Engine to understand the diffs created by the Frontend and Backend agents to synthesize accurate documentation.

### 6. Automation Agent
- **Role:** The workflow optimizer and DevOps engineer.
- **Responsibilities:** Manages the deployment pipelines, CI/CD configurations, and the PEN.GUIN ecosystem's internal automation scripts. It can write GitHub Actions, configure Dockerfiles, and automate repetitive repository maintenance tasks.
- **Interaction:** Works closely with the system's `Workflow Engine` to ensure the transition from generated artifact to deployed software is seamless.

## How the Routing System Assigns Tasks

The `Agent Router` uses a deterministic matching algorithm to decide which agent executes a specific task, ensuring tasks are handled by the correct specialist.

1. **Task Classification Tags:** When the `Task Classifier` breaks down an objective, it assigns domain tags (e.g., `[UI, React]`, `[Database, SQL]`, `[Design, Planning]`).
2. **Competency Matching:** The Router compares these task tags against the registered competency matrix of each Agent Type. 
   - A task tagged `[UI]` instantly routes to the **Frontend Agent**.
   - A task tagged `[Design]` routes to the **Architect Agent**.
3. **Adaptive Weighting:** If multiple agents *could* theoretically handle a task, the Router consults the `Decision Memory`. It looks at historical performance metrics (Execution Time, Review Scores) to select the agent with the highest probability of success for that specific type of work.
4. **Lifecycle Phase Alignment:** The Router also considers the current phase of the workflow. If a piece of code was just generated by the Backend Agent, the next logical step in the pipeline automatically routes the artifact to the **Review Agent**, overriding standard domain tags.

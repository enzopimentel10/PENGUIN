# Workspace Environment Architecture

The `/workspace` directory serves as the operational theater for all AI agents within the PEN.GUIN ecosystem. It is the structured sandbox where objectives are transformed into concrete actions and verifiable outputs.

## Directory Structure & Purpose

The workspace is divided into three core directories:

### 1. `tasks/`
**Purpose:** This folder acts as the inbox and queue for all pending and active agent tasks.
**Details:** When the `Task Classifier` breaks down a user objective into sub-tasks, it generates structured JSON or YAML files inside this directory (e.g., `task-001-frontend-ui.json`). These files contain all the necessary parameters, context boundaries, and success criteria for a specific agent to begin work.

### 2. `sessions/`
**Purpose:** This folder tracks the real-time execution, logs, and contextual state of active workflows.
**Details:** A session represents a continuous thread of execution toward a specific objective. When an agent begins a task, a corresponding session file (or directory) is created here. It stores the agent's scratchpad, temporary file modifications, tool execution logs (like stdout/stderr from the `Task Executor`), and step-by-step reasoning. It is the "live memory" of an ongoing process, allowing the `Workflow Engine` to pause, resume, or rollback executions if errors occur.

### 3. `artifacts/`
**Purpose:** This folder is the secure storage for the final, validated outputs generated by the agents.
**Details:** Artifacts are the tangible results of completed tasks. This can include compiled binaries, generated database schemas, newly written code files (before they are merged into the main repository tree), test coverage reports, or architectural diagrams. Only output that has passed all validation gates (testing, security review) is promoted to the artifacts folder.

## Agent Workflow: From Tasks to Artifacts

### 1. Receiving Tasks
Agents do not spontaneously generate work; they listen to the `tasks/` directory via the `Workflow Engine`. When an agent is assigned a task by the `Agent Router`, it reads the corresponding file in `tasks/` to understand its objective. The AI Kernel then provisions the agent with the necessary skills and maps the required repository context.

### 2. Tracking Execution in Sessions
As the agent executes the task, it operates within the context of a "Session." The `sessions/` directory acts as a state machine. Every tool call, every file read, and every code modification is logged into the active session file. If the Frontend Agent needs to pass an API contract to the Backend Agent, the Context Engine facilitates this handoff by writing intermediate states into the session, ensuring smooth continuation of the workflow. If an agent fails, the session log provides the exact error trace needed for the "Self-Healing" protocol to trigger a retry.

### 3. Storing Results as Artifacts
Once a task is complete and the agent determines success (e.g., tests pass), the final outputs are serialized and saved to the `artifacts/` directory. For example, if the Documentation Agent generates a new API spec, it saves `api-v1-spec.md` to `artifacts/`. The `Workflow Engine` then inspects the artifact. If it meets system requirements, the engine may automatically commit the artifact into the main codebase (e.g., moving it to `/docs/api/`) and mark the original task in `tasks/` as completed.

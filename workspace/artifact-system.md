# Artifact System

The Artifact System is the authoritative ledger of all finalized outputs produced by AI agents within the PEN.GUIN ecosystem. Artifacts are the tangible, verifiable results of completed tasks, stored securely before being promoted into the main repository or utilized in subsequent workflows.

## Types of Artifacts

Agents produce various types of artifacts depending on their specialized roles and the nature of the task:

- **Generated Code:** The most common artifact. This includes source code files (`.js`, `.py`, `.rs`), stylesheets, and markup templates generated by the Frontend or Backend Agents. These are often stored as diffs or complete isolated files before integration.
- **Analysis Results:** Outputs from research phases. Examples include dependency graphs from the `Architecture Agent`, database schema models, or algorithmic profiling reports.
- **Review Reports:** Structured documents generated by the `Review Agent` or `Security Agent`. These contain linting summaries, cyclomatic complexity scores, vulnerability assessments (SAST outputs), and required remediation steps.
- **Logs & Test Traces:** Immutable records of execution. This includes the `stdout`/`stderr` from build processes, test coverage reports from the `Test Agent`, and step-by-step reasoning logs from an agent's cognitive loop.

## Storage and Indexing

Artifacts are not dumped into a single unstructured folder. They are systematically managed to ensure traceability and easy retrieval.

### Storage Structure
Artifacts are stored within the `/workspace/artifacts/` directory. The structure is dynamically partitioned based on time, domain, or project module:
- `/workspace/artifacts/<timestamp_or_run_id>/code/`
- `/workspace/artifacts/<timestamp_or_run_id>/reports/`
- `/workspace/artifacts/<timestamp_or_run_id>/logs/`

### Indexing
To prevent the artifacts folder from becoming a black box, the `Workflow Engine` maintains a centralized `artifact-index.json` (or a lightweight SQLite table in the `Memory Model`). 
When an agent generates an artifact, the engine registers it in the index with the following metadata:
- `artifact_id`: A unique UUID.
- `type`: The category (code, report, log).
- `file_path`: Absolute or relative path to the physical file.
- `authoring_agent`: The specific agent (e.g., `frontend-agent`) that generated it.
- `timestamp`: Creation time.
- `status`: e.g., `Pending Review`, `Verified`, `Merged`, `Rejected`.

## Linking to Tasks and Sessions

The true power of the Artifact System lies in its relational traceability. Every artifact is explicitly linked to the causal chain that created it.

1. **Task Linkage:** When the `Workflow Engine` saves an artifact, it injects the `task_id` (from the `/workspace/tasks/` queue) into the artifact's metadata index. This allows a user or a Review Agent to trace a piece of code exactly back to the specific objective that demanded its creation.
2. **Session Linkage:** Furthermore, the artifact is linked to the `session_id` (from the `/workspace/sessions/` directory). If an artifact is found to be flawed during a later review, the system can pull up the exact session log to analyze the agent's step-by-step reasoning and tool calls. This allows the Kernel to understand *why* the agent made a mistake, facilitating better error recovery and system learning.

By maintaining strict relational links (Objective -> Task -> Session -> Artifact), the PEN.GUIN ecosystem ensures absolute observability over the AI development pipeline.
